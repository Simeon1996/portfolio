## CLI Commands & Flags

### Startup commands

| Command | What it does |
|---------|-------------|
| `claude` | Start interactive REPL |
| `claude "query"` | Start with initial prompt |
| `claude -p "query"` | Non-interactive / print mode (exits after response) |
| `claude -c` | Continue most recent conversation |
| `claude -r "<session-id>"` | Resume specific session by ID |
| `claude update` | Update to latest version |

### All flags

| Flag | Short | Purpose |
|------|-------|---------|
| `--model <name>` | | Model: `sonnet`, `opus`, `haiku`, or full model ID |
| `--print` | `-p` | Non-interactive, output to stdout |
| `--output-format` | | `text`, `json`, or `stream-json` |
| `--permission-mode <mode>` | | `default` / `acceptEdits` / `plan` / `auto` / `bypassPermissions` |
| `--dangerously-skip-permissions` | | Skip all permission prompts |
| `--allowedTools <tools>` | | Pre-approve tools (space-separated permission rules) |
| `--disallowedTools <tools>` | | Remove tools from availability |
| `--add-dir <path>` | | Grant file access to an additional directory |
| `--system-prompt <text>` | | Replace the entire system prompt |
| `--append-system-prompt <text>` | | Append to the default system prompt |
| `--effort <level>` | | `low` / `medium` / `high` / `max` (Opus only) |
| `--worktree` | `-w` | Start in an isolated git worktree |
| `--bare` | | Skip auto-discovery for faster startup |
| `--mcp-config <file>` | | Load MCP servers from a JSON file |
| `--strict-mcp-config` | | Only use servers from `--mcp-config` |
| `--name <name>` | `-n` | Set a session display name |
| `--verbose` | | Enable verbose logging |

### Pipeline and scripting examples

```bash
# Non-interactive, pipe a file as input
claude -p "Summarize this file" < main.py

# JSON output for scripting
claude -p "List all functions in this file" --output-format json < app.py

# Stream JSON tokens as they are generated
claude -p "Write tests for auth.ts" --output-format stream-json

# Skip permissions in a container (isolated environments only)
claude --dangerously-skip-permissions -p "Run full test suite and fix failures"

# Pre-approve specific tools
claude --allowedTools "Bash(git *)" --allowedTools "Read"

# Use a specific model
claude --model claude-opus-4-6

# Append context without replacing the system prompt
claude --append-system-prompt "Always respond in under 200 words."
```

## CLAUDE.md Reference

### File locations

All files in the list below are loaded and concatenated — there is no override. Claude sees all of them simultaneously.

```
~/.claude/CLAUDE.md                                            # User-level — applies to all projects
./CLAUDE.md  or  ./.claude/CLAUDE.md                          # Project-level — commit this to git
./CLAUDE.local.md                                              # Local personal — gitignore this
/etc/claude-code/CLAUDE.md                                     # Managed / org-level (Linux)
~/Library/Application Support/ClaudeCode/CLAUDE.md            # Managed / org-level (macOS)
```

### Loading order (walking up from cwd)

```
./CLAUDE.md + ./CLAUDE.local.md
../CLAUDE.md + ../CLAUDE.local.md
../../CLAUDE.md ...  (continues up to filesystem root)
~/.claude/CLAUDE.md  (user level, always last)
```

> Subdirectory CLAUDE.md files load on-demand when those directories are accessed, not at session start.

### Typical structure

```markdown
# Project Name

## Build & Test Commands
- Build: `npm run build`
- Test: `npm test`
- Lint: `npm run lint`
- Type check: `npx tsc --noEmit`

## Architecture
Brief description of how the project is structured —
key directories, main entry points, data flow.

## Code Style
- TypeScript strict mode is required
- Prefer `const` over `let`; never use `var`
- No `any` types — use `unknown` and narrow explicitly
- All async functions must handle rejections

## Key Conventions
- All API endpoints return `{ data, error }` shape
- Database access goes through `src/db/` only — no raw SQL elsewhere
- Feature flags live in `src/config/flags.ts`
```

### Importing other files

```markdown
@path/to/shared-rules.md
@../team/code-standards.md
```

### Path-scoped rules via `.claude/rules/`

```yaml
---
# .claude/rules/api-rules.md
paths:
  - "src/api/**/*.ts"
  - "src/routes/**"
---
# API Rules
- Validate all inputs at the route level before they reach service logic
- Use standard `{ data, error }` response shape on every endpoint
- Never return raw database rows — always map through a response schema
```

### Tips

- Keep each file under ~200 lines. Split by concern, not by length.
- HTML comments `<!-- notes -->` are stripped from context — use them for maintainer notes.
- `CLAUDE.local.md` appends after `CLAUDE.md` at the same directory level.
- If other tools use `AGENTS.md`, import it with `@AGENTS.md` and add Claude-specific instructions below the import.

## settings.json Reference

### File locations (highest to lowest priority)

```
.claude/settings.local.json      # Project local (gitignore this)
.claude/settings.json            # Project shared (commit to git)
~/.claude/settings.json          # User-level defaults
```

### Full structure

```json
{
  "model": "claude-opus-4-6",
  "defaultMode": "acceptEdits",
  "effort": "high",
  "autoMemoryEnabled": true,
  "autoMemoryDirectory": "~/.claude/projects",
  "additionalDirectories": ["/path/to/shared/libs"],
  "claudeMdExcludes": ["**/vendor/CLAUDE.md"],
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(npm run test *)",
      "Bash(npm run lint)",
      "Read",
      "Edit",
      "mcp__puppeteer__*"
    ],
    "deny": [
      "Read(./.env)",
      "Bash(git push --force *)",
      "Bash(rm -rf *)"
    ],
    "disableBypassPermissionsMode": "disable",
    "disableAutoMode": "disable"
  },
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint --fix"
          }
        ]
      }
    ]
  }
}
```

### Permission rule syntax

```
Bash(git log *)          # Allow git log with any args
Bash(npm run test *)     # Allow npm run test with any args
Read(./.env)             # Scoped to a specific file
Edit(src/**)             # Scoped to a glob pattern
mcp__server__tool        # A specific MCP tool
mcp__server__*           # All tools from one MCP server
Read                     # All reads, no path restriction
```

## Hooks System

### Hook events in lifecycle order

```
SessionStart           → session begins or resumes
InstructionsLoaded     → CLAUDE.md files loaded into context
UserPromptSubmit       → before Claude processes the user's prompt
  PreToolUse           → before each tool call (can block the call)
  PostToolUse          → after a successful tool call
  PostToolUseFailure   → after a failed tool call
Stop                   → Claude finishes responding
StopFailure            → turn ends due to an API error
SessionEnd             → session terminates
```

### Hook configuration in settings.json

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "./.claude/hooks/pre-bash.sh"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint"
          }
        ]
      }
    ],
    "Stop": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "./.claude/hooks/notify.sh"
          }
        ]
      }
    ]
  }
}
```

### Hook types

```json
{ "type": "command", "command": "./script.sh", "async": false }
{ "type": "http", "url": "https://api.example.com/hook", "timeout": 30 }
{ "type": "prompt", "prompt": "Is this safe? $ARGUMENTS" }
{ "type": "agent", "agent": "Explore" }
```

### Hook exit codes

| Exit code | Meaning |
|-----------|---------|
| `0` | Success — process any JSON on stdout |
| `2` | Block the tool call — write reason to stderr |
| Other | Non-blocking error — shown in transcript, does not block |

### Blocking a tool call (exit code 2)

```bash
#!/bin/bash
# .claude/hooks/pre-bash.sh
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // ""')

if echo "$COMMAND" | grep -qE 'rm -rf|DROP TABLE|DELETE FROM'; then
  echo "Blocked: destructive command detected" >&2
  exit 2
fi
exit 0
```

### Hook stdout JSON (optional return payload)

```json
{
  "continue": true,
  "decision": "allow",
  "reason": "Command is safe",
  "additionalContext": "Reminder: use --dry-run first",
  "suppressOutput": false
}
```

### Matcher patterns

| Pattern | Matches |
|---------|---------|
| `"*"` or omit field | All events |
| `"Bash"` | Exact tool name |
| `"Edit\|Write"` | Pipe-separated list (either/or) |
| `"^mcp__"` | JavaScript regex — tools from any MCP server |
| `"mcp__.*"` | Same as above, alternate form |

### Disable all hooks

```json
{ "disableAllHooks": true }
```

## Permission Modes

| Mode | Auto-approves | Blocks | Use case |
|------|--------------|--------|----------|
| `default` | Nothing (prompts on first use per tool) | Nothing | General development |
| `acceptEdits` | File edits + filesystem operations | Nothing | Fast iteration on code |
| `plan` | Nothing | All write operations | Code review and analysis only |
| `auto` | Most operations (background safety checks) | Risky operations | CI and automation |
| `bypassPermissions` | Everything | Protected directories | Containers and isolated VMs only |

### Set via settings.json

```json
{ "defaultMode": "acceptEdits" }
```

### Set via CLI

```bash
claude --permission-mode plan
claude --dangerously-skip-permissions
```

> Cycle modes mid-session with `Shift+Tab`.

### Protected directories (always blocked, even in bypassPermissions)

`.git` `.claude` `.vscode` `.idea` `.husky`

## MCP Configuration

### Config file locations

```
~/.claude.json            # User-level — applies to all projects
.claude/mcp.json          # Project-level — scoped to this repo
```

### Full config example

```json
{
  "mcpServers": {
    "postgres": {
      "command": "python",
      "args": ["-m", "mcp_server_postgres"],
      "env": { "DATABASE_URL": "${DATABASE_URL}" }
    },
    "github": {
      "command": "node",
      "args": ["./github-mcp/index.js"],
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }
    },
    "remote-api": {
      "url": "https://my-mcp-server.example.com/sse",
      "headers": { "Authorization": "Bearer ${API_TOKEN}" }
    },
    "filesystem": {
      "command": "uvx",
      "args": ["mcp-server-filesystem", "--allowed-paths", "/tmp"]
    }
  }
}
```

### CLI flags

```bash
claude --mcp-config ./mcp.json                       # Load MCP servers from a file
claude --mcp-config ./mcp.json --strict-mcp-config   # Only use servers in this file
```

### Permission rules for MCP tools

```json
{
  "permissions": {
    "allow": ["mcp__github__*"],
    "deny":  ["mcp__github__delete_repo"]
  }
}
```

## Memory System

### Two types

| | CLAUDE.md | Auto Memory |
|--|----------|-------------|
| Written by | You | Claude automatically |
| Contains | Rules, conventions, commands | Learnings, patterns, discoveries |
| Location | Project / user / org directories | `~/.claude/projects/<hash>/memory/` |
| View / edit | `/memory` command | `/memory` command |

### Auto memory config

```json
{
  "autoMemoryEnabled": true,
  "autoMemoryDirectory": "~/my-memory"
}
```

### Disable via environment variable

```bash
CLAUDE_CODE_DISABLE_AUTO_MEMORY=1 claude
```

> Auto memory is scoped to git root, not cwd. Multiple subdirectories within the same repo share one memory directory. Different repos get different directories.

## Shell (`!`) Prefix

**Syntax:** `! <any shell command>`

Runs the command in your shell, captures the output, and injects both the command and result into the conversation. Claude sees what you ran and what it returned.

```
! git diff HEAD~1
! npm test 2>&1 | tail -30
! cat src/auth.ts
! node --version && npm --version
! curl -s http://localhost:3000/health | jq .
```

### Filter output — context fills fast

| | Command | Why |
|--|---------|-----|
| Good | `! npm test \| tail -20` | Keeps only the summary |
| Bad | `! npm test` | Dumps every line of test output |
| Good | `! cat access.log \| grep ERROR` | Only relevant lines |
| Bad | `! cat access.log` | Potentially megabytes of data |
| Good | `! env \| grep -v SECRET` | Filters credentials |
| Bad | `! env` | Exposes all environment secrets to context |

## Custom Agents & Skills

### Custom agent config

Place at `.claude/agents/<name>/AGENT.md`:

```yaml
---
name: code-reviewer
description: Specialized code review agent with access to diff and linting tools
model: claude-opus-4-6
effort: high
allowedTools: Bash(git diff *) Bash(npm run lint) Read Grep
disallowedTools: Write Edit
---

You are a senior code reviewer. Focus on:
- Security vulnerabilities
- Performance regressions
- Test coverage gaps

Always cite file paths and line numbers.
```

### Invoke an agent

```
@code-reviewer  review the changes in src/auth/
claude --agent code-reviewer
```

### Custom skill

Place at `.claude/skills/<name>/SKILL.md`:

```yaml
---
name: deploy-check
description: Pre-deployment verification checklist
allowedTools: Bash(npm run test) Bash(npm run build) Read
context: fork
---

Run the full deployment verification:
1. `npm run test` — all tests must pass
2. `npm run build` — must complete without errors
3. Check that CHANGELOG.md has an entry for this version
4. Verify .env.example is up to date

Report: PASS or FAIL with specific issues listed.
```

### Invoke a skill

```
/deploy-check
```

### Preload skills into an agent

```yaml
---
name: senior-dev
skills:
  - deploy-check
  - simplify
  - commit
---
```

### Skill auto-invocation (path-scoped)

```yaml
---
name: api-guard
paths:
  - "src/api/**/*.ts"
user-invocable: false
---
Whenever a file in src/api/ is edited, verify it follows the API contract
and that all inputs are validated at the route level.
```

> `user-invocable: false` means the skill runs automatically based on the `paths` pattern — it does not appear as a slash command.

## Common Gotchas

**1. CLAUDE.md files are additive, not overriding.**
All CLAUDE.md files in the directory hierarchy are loaded and concatenated. There is no concept of "override." If two files give conflicting instructions, Claude sees both and has to reconcile them. Keep instructions consistent across levels, or use path-scoped rules to avoid conflicts.

**2. `bypassPermissions` does not mean truly unlimited access.**
`.git`, `.claude`, `.vscode`, `.idea`, and `.husky` are always protected regardless of mode. Use `bypassPermissions` only inside isolated containers or VMs — never in a shared development environment.

**3. Auto memory is scoped to git root, not cwd.**
Memory files live under a hash derived from your git repo root. If you run Claude from different subdirectories in the same repo, they share one memory directory. Switching to a different repo switches to a different memory directory automatically.

**4. `/compact` loses non-CLAUDE.md context.**
After `/compact`, the project-root CLAUDE.md re-injects. Subdirectory CLAUDE.md files reload on-demand. Any instructions that only existed in the conversation — not in any file — are gone. Move important conventions to CLAUDE.md before they matter.

**5. Hook `matcher` is a regex, not a glob.**
`"Edit|Write"` works as expected (alternation). `"Edit*"` does not do what you expect — the `*` is a regex quantifier meaning "zero or more `t` characters," not a glob wildcard. Use `"Edit|Write|Notebook"` or `"^(Edit|Write)"` when matching multiple tools.

**6. `--print` and interactive mode are mutually exclusive.**
`claude -p "query"` exits after the first response. Session continuation (`-c`, `-r`) requires interactive mode. Do not combine `-p` with workflows that expect back-and-forth conversation.

**7. MCP HTTP hooks require explicit env var allowlisting.**
```json
{ "type": "http", "url": "...", "allowedEnvVars": ["GITHUB_TOKEN", "DATABASE_URL"] }
```
Without `allowedEnvVars`, environment variables are not forwarded to HTTP hook endpoints. The request will succeed but arrive without the credentials your endpoint expects.

## Key Takeaways

- **CLAUDE.md files are additive.** Every file in the hierarchy is loaded — project, user, and org level simultaneously. Conflicts are not resolved automatically; Claude sees all instructions and must reconcile them.
- **Hooks are the automation layer.** `PreToolUse` with exit code `2` is a hard block. `PostToolUse` on `Edit|Write` is your auto-lint trigger. `Stop` is your notification hook. These three cover most production automation needs.
- **Permission modes are a dial, not a toggle.** `default` prompts once per tool. `acceptEdits` auto-approves file changes. `plan` blocks all writes. `auto` runs with background safety checks. `bypassPermissions` removes prompts entirely — containers only.
- **`--print` is the scripting primitive.** Combine it with `--output-format json` or `stream-json` and pipe it into your own tooling. This is how you build Claude into CI, cron jobs, and automation pipelines without a custom API integration.
- **The `!` prefix is for targeted, filtered output.** Pipe and filter everything — raw `npm test` output or a full `env` dump will burn your context window and potentially leak secrets into the conversation.
