## Quick-Reference Card

The three MCP primitives, when to use each, and their Python decorator syntax in one table.

| Primitive | What it does | When to use it | Python decorator |
|---|---|---|---|
| **Tool** | Executable function the model can call. Returns structured output. Has side effects or queries live data. | Any action: search, read DB, call API, write file, run code | `@mcp.tool()` |
| **Resource** | Read-only data exposed via a URI. The model (or client) fetches it like a file or URL. | Static schemas, config blobs, database snapshots, file contents | `@mcp.resource("scheme://path")` |
| **Prompt** | Reusable message template with named arguments. Produces a structured prompt the host injects. | Standardized instructions, system prompts, user-facing templates | `@mcp.prompt()` |

> **Rule of thumb:** if the model needs to *do* something, it's a Tool. If the model needs to *read* something on demand, it's a Resource. If you want the host to inject a consistent prompt, it's a Prompt.

## Transport Types

Three transports exist. The choice is determined by deployment topology, not preference.

| Transport | When to use | Direction | Claude Desktop support |
|---|---|---|---|
| **stdio** | Local process launched by the host. Zero network config. | Bidirectional over stdin/stdout | Yes — primary mode |
| **HTTP + SSE** | Remote server, cloud deployment, multi-client. Host connects via HTTP; server streams events over SSE. | Client→Server: HTTP POST. Server→Client: SSE | Yes (as of 2025 spec) |
| **WebSocket** | Long-lived bidirectional sessions; low-latency streaming. Less common in the current SDK. | Full duplex | Not natively in Claude Desktop |

### stdio config (claude_desktop_config.json)

```json
{
  "mcpServers": {
    "my-local-server": {
      "command": "python",
      "args": ["/absolute/path/to/server.py"],
      "env": {
        "DATABASE_URL": "postgresql://localhost/mydb"
      }
    }
  }
}
```

### HTTP/SSE server startup (Python)

```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("my-server")

# ...register tools, resources, prompts...

if __name__ == "__main__":
    # Run over HTTP+SSE on port 8000
    mcp.run(transport="sse", host="0.0.0.0", port=8000)
```

### stdio server startup (Python)

```python
if __name__ == "__main__":
    # Default transport — launched by host as a subprocess
    mcp.run(transport="stdio")
```

## TypeScript Full Server Skeleton

Same structure, TypeScript, using `@modelcontextprotocol/sdk`.

```typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

// ── Server init ──────────────────────────────────────────────────────────────
const server = new McpServer({
  name: "my-server",
  version: "1.0.0",
});

// ── Tool ──────────────────────────────────────────────────────────────────────
server.tool(
  "search_records",
  "Search records by query string. Returns a list of matches with id, title, and score.",
  {
    query: z.string().describe("Full-text search query"),
    limit: z.number().int().min(1).max(100).default(10).describe("Max results"),
  },
  async ({ query, limit }) => {
    // Replace with real implementation
    const results = [{ id: 1, title: `Result for '${query}'`, score: 0.95 }];
    return {
      content: [
        {
          type: "text",
          text: JSON.stringify(results.slice(0, limit), null, 2),
        },
      ],
    };
  }
);

// ── Tool with side effects ─────────────────────────────────────────────────────
server.tool(
  "delete_record",
  "WRITE OPERATION: Permanently deletes a record by ID. This cannot be undone.",
  {
    record_id: z.string().describe("ID of the record to delete"),
  },
  async ({ record_id }) => {
    if (!record_id) {
      throw new Error("record_id must not be empty");
    }
    // Replace with real delete logic
    return {
      content: [
        {
          type: "text",
          text: JSON.stringify({ deleted: record_id, status: "ok" }),
        },
      ],
    };
  }
);

// ── Resource ──────────────────────────────────────────────────────────────────
server.resource(
  "config://schema",
  "config://schema",
  { description: "JSON schema for records in this server" },
  async (_uri) => ({
    contents: [
      {
        uri: "config://schema",
        mimeType: "application/json",
        text: '{"type":"object","properties":{"id":{"type":"integer"},"title":{"type":"string"}}}',
      },
    ],
  })
);

// ── Prompt ────────────────────────────────────────────────────────────────────
server.prompt(
  "analyze_record",
  "Reusable prompt template for analyzing a record.",
  {
    record_title: z.string().describe("Title of the record to analyze"),
    context: z.string().optional().describe("Optional additional context"),
  },
  ({ record_title, context }) => {
    let text = `Analyze the following record and summarize its key attributes: '${record_title}'.`;
    if (context) text += `\n\nAdditional context: ${context}`;
    return {
      messages: [{ role: "user", content: { type: "text", text } }],
    };
  }
);

// ── Entrypoint ────────────────────────────────────────────────────────────────
async function main() {
  const transport = new StdioServerTransport();
  await server.connect(transport); // do not forget this await
  console.error("my-server running on stdio");
}

main().catch((err) => {
  console.error("Fatal:", err);
  process.exit(1);
});
```

## Tool Patterns

### Pattern 1 — Simple read-only tool, returns JSON

```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("read-demo")

@mcp.tool()
async def get_user(user_id: str) -> dict:
    """
    Fetch a user record by ID. Read-only. No side effects.

    Args:
        user_id: UUID string identifying the user.

    Returns:
        User object with id, name, and email fields.
    """
    # Replace with real DB call
    return {"id": user_id, "name": "Alice", "email": "alice@example.com"}
```

### Pattern 2 — Tool with input validation (Pydantic)

```python
from mcp.server.fastmcp import FastMCP
from pydantic import BaseModel, Field, field_validator

mcp = FastMCP("validation-demo")

class CreateItemInput(BaseModel):
    name: str = Field(..., min_length=1, max_length=200)
    price: float = Field(..., gt=0)
    tags: list[str] = Field(default_factory=list, max_length=10)

    @field_validator("name")
    @classmethod
    def name_must_not_be_blank(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("name must not be blank")
        return v.strip()

@mcp.tool()
async def create_item(name: str, price: float, tags: list[str] = []) -> dict:
    """
    Create a new inventory item. Validates name, price, and tag list.

    Args:
        name: Item name (1–200 characters, non-blank).
        price: Unit price in USD. Must be greater than 0.
        tags: Optional list of tag strings (max 10).

    Returns:
        Created item with assigned id.
    """
    # Pydantic validates on construction — raises McpError on failure
    validated = CreateItemInput(name=name, price=price, tags=tags)
    return {"id": "item_abc123", "name": validated.name, "price": validated.price}
```

### Pattern 3 — Tool with side effects (explicit warning in docstring)

```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("write-demo")

@mcp.tool()
async def send_email(to: str, subject: str, body: str) -> dict:
    """
    WRITE OPERATION: Sends an email to the specified recipient.

    WARNING: This action delivers a real email and cannot be undone.
    Do not call this tool unless the user has explicitly confirmed the recipient,
    subject, and body.

    Args:
        to: Recipient email address.
        subject: Email subject line.
        body: Plain-text email body.

    Returns:
        Delivery status with message_id.
    """
    # Replace with real email client call (e.g., boto3 SES, SendGrid)
    return {"status": "sent", "message_id": "msg_xyz789", "to": to}
```

> Side-effect tools should always carry a `WRITE OPERATION:` label in the first line of the docstring and explicit user-confirmation language. This surfaces in the model's tool description and conditions its calling behavior.

### Pattern 4 — Async tool with timeout

```python
import asyncio
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("timeout-demo")

async def _fetch_external_data(endpoint: str) -> dict:
    """Internal helper — wraps actual HTTP call."""
    await asyncio.sleep(0.1)  # replace with real aiohttp / httpx call
    return {"endpoint": endpoint, "data": "..."}

@mcp.tool()
async def fetch_with_timeout(endpoint: str, timeout_seconds: float = 10.0) -> dict:
    """
    Fetch data from an external endpoint with a hard timeout.

    Args:
        endpoint: Full URL of the endpoint to fetch.
        timeout_seconds: Max wait time in seconds (default 10, max 30).

    Returns:
        Response data dict, or raises TimeoutError if deadline exceeded.
    """
    timeout_seconds = min(timeout_seconds, 30.0)
    try:
        result = await asyncio.wait_for(
            _fetch_external_data(endpoint),
            timeout=timeout_seconds,
        )
        return result
    except asyncio.TimeoutError:
        raise TimeoutError(f"Request to {endpoint} exceeded {timeout_seconds}s timeout")
```

## Resource Patterns

### Pattern 1 — Static resource (fixed content, e.g. a schema)

```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("resource-demo")

@mcp.resource("config://db-schema")
def get_db_schema() -> str:
    """
    Returns the current database schema as a JSON string.
    Static — content does not change at runtime.
    """
    return """{
  "tables": {
    "users": {
      "columns": ["id", "name", "email", "created_at"]
    },
    "orders": {
      "columns": ["id", "user_id", "total", "status", "created_at"]
    }
  }
}"""
```

Access URI: `config://db-schema`

### Pattern 2 — Dynamic resource with URI template

```python
from mcp.server.fastmcp import FastMCP
import json

mcp = FastMCP("resource-demo")

# The {table_name} segment is extracted and passed as a parameter
@mcp.resource("db://tables/{table_name}")
async def get_table_info(table_name: str) -> str:
    """
    Returns column metadata for the requested database table.

    URI template: db://tables/<table_name>
    Example: db://tables/users
    """
    # Replace with real schema introspection
    schemas = {
        "users": {"columns": ["id", "name", "email"], "row_count": 14200},
        "orders": {"columns": ["id", "user_id", "total"], "row_count": 87400},
    }
    if table_name not in schemas:
        raise ValueError(f"Unknown table: {table_name}")
    return json.dumps(schemas[table_name])
```

Access URI: `db://tables/users` or `db://tables/orders`

> URI scheme names are arbitrary strings — they have no connection to real URL schemes. Pick something descriptive and consistent across your server (`db://`, `config://`, `file://`, `api://`). The client resolves them through your server, not through DNS.

## Prompt Patterns

### Reusable prompt template with arguments

```python
from mcp.server.fastmcp import FastMCP
from mcp.types import PromptMessage, TextContent

mcp = FastMCP("prompt-demo")

@mcp.prompt()
def code_review(
    language: str,
    code: str,
    focus: str = "correctness, performance, and security",
) -> list[PromptMessage]:
    """
    Reusable code review prompt. Injects language, code, and a configurable focus area.

    Args:
        language: Programming language (e.g. "Python", "TypeScript").
        focus: Comma-separated review focus areas.
        code: The code block to review.
    """
    system_text = (
        f"You are a senior {language} engineer. "
        f"Review the following code focusing on: {focus}. "
        "Be specific. Cite line numbers where relevant. "
        "Prioritize issues by severity: critical, major, minor."
    )
    user_text = f"```{language.lower()}\n{code}\n```"

    return [
        PromptMessage(role="user", content=TextContent(type="text", text=system_text)),
        PromptMessage(role="user", content=TextContent(type="text", text=user_text)),
    ]
```

## Client Connection Patterns

All three connection modes use the same `ClientSession` API once connected. Only the transport construction differs.

### stdio client

```python
import asyncio
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

async def main():
    params = StdioServerParameters(
        command="python",
        args=["/absolute/path/to/server.py"],
        env={"DATABASE_URL": "postgresql://localhost/mydb"},
    )

    async with stdio_client(params) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()

            # List available tools
            tools_result = await session.list_tools()
            for tool in tools_result.tools:
                print(f"Tool: {tool.name} — {tool.description}")

            # Call a tool
            result = await session.call_tool(
                "search_records",
                arguments={"query": "machine learning", "limit": 5},
            )
            print(result.content)

asyncio.run(main())
```

### HTTP/SSE client

```python
import asyncio
from mcp import ClientSession
from mcp.client.sse import sse_client

async def main():
    server_url = "http://localhost:8000/sse"

    async with sse_client(server_url) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()

            # List resources
            resources_result = await session.list_resources()
            for resource in resources_result.resources:
                print(f"Resource: {resource.uri}")

            # Read a resource
            content = await session.read_resource("config://db-schema")
            print(content.contents[0].text)

            # Call a tool
            result = await session.call_tool(
                "get_user",
                arguments={"user_id": "usr_123"},
            )
            print(result.content)

asyncio.run(main())
```

### Listing tools and calling programmatically

```python
import asyncio
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

async def call_tool(tool_name: str, args: dict) -> str:
    params = StdioServerParameters(command="python", args=["server.py"])
    async with stdio_client(params) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()
            result = await session.call_tool(tool_name, arguments=args)
            # result.content is a list of ContentBlock objects
            return "\n".join(
                block.text for block in result.content if hasattr(block, "text")
            )

if __name__ == "__main__":
    output = asyncio.run(call_tool("search_records", {"query": "python"}))
    print(output)
```

## Error Handling Reference

### MCP error codes

| Code | Constant | Meaning | When to raise |
|---|---|---|---|
| `-32700` | `PARSE_ERROR` | Invalid JSON received | Transport/SDK handles this automatically |
| `-32600` | `INVALID_REQUEST` | Request object malformed | SDK handles; rarely raised manually |
| `-32601` | `METHOD_NOT_FOUND` | Requested method doesn't exist | SDK handles automatically |
| `-32602` | `INVALID_PARAMS` | Invalid method parameters | Raise when input fails validation |
| `-32603` | `INTERNAL_ERROR` | Unexpected server error | Catch-all for unhandled exceptions |
| `-32000` | `SERVER_ERROR` (base) | Server-defined error range | Use for domain-specific errors |

### Raising errors in Python

```python
from mcp.server.fastmcp import FastMCP
from mcp.shared.exceptions import McpError
from mcp.types import ErrorData, INVALID_PARAMS, INTERNAL_ERROR

mcp = FastMCP("error-demo")

@mcp.tool()
async def divide(numerator: float, denominator: float) -> float:
    """Divide two numbers. Raises on zero denominator."""
    if denominator == 0:
        raise McpError(
            ErrorData(
                code=INVALID_PARAMS,
                message="denominator must not be zero",
            )
        )
    try:
        return numerator / denominator
    except Exception as exc:
        raise McpError(
            ErrorData(
                code=INTERNAL_ERROR,
                message=f"Unexpected error during division: {exc}",
            )
        )
```

### Raising errors in TypeScript

```typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { McpError, ErrorCode } from "@modelcontextprotocol/sdk/types.js";
import { z } from "zod";

const server = new McpServer({ name: "error-demo", version: "1.0.0" });

server.tool(
  "divide",
  "Divide two numbers. Raises McpError on zero denominator.",
  {
    numerator: z.number(),
    denominator: z.number(),
  },
  async ({ numerator, denominator }) => {
    if (denominator === 0) {
      throw new McpError(
        ErrorCode.InvalidParams,
        "denominator must not be zero"
      );
    }
    try {
      const result = numerator / denominator;
      return { content: [{ type: "text", text: String(result) }] };
    } catch (err) {
      throw new McpError(
        ErrorCode.InternalError,
        `Unexpected error: ${err}`
      );
    }
  }
);
```

> Always raise `McpError` (not a plain `Exception` or `Error`) when you want the client to receive a structured error response with a code. Unhandled exceptions are caught by the SDK and mapped to `INTERNAL_ERROR`, but without a useful message.

## Common Gotchas

**1. Using stdio transport for a server you deployed remotely.**
stdio works by spawning a subprocess. If your server lives on a remote host (container, Lambda, VM), the host cannot `exec` it. Switch to `transport="sse"` server-side and use the `url` field in `claude_desktop_config.json`. This fails silently — the server just never appears in Claude Desktop.

**2. Missing `await server.connect(transport)` in TypeScript.**
The `McpServer` constructor does not begin listening. You must call `await server.connect(transport)` explicitly. Omitting the `await` means the process exits before any client can negotiate. The server appears to start with no error, then immediately terminates.

**3. Zod schema versus JSON Schema mismatch in TypeScript.**
The SDK accepts Zod schemas for tool inputs and converts them internally. Do not pass raw JSON Schema objects where Zod is expected — the types look similar but the runtime behavior differs. If a field silently accepts anything, check whether you passed a plain object instead of `z.object({...})`.

**4. Resource URI format must match the registered template exactly.**
A resource registered as `db://tables/{table_name}` will not match a client request for `db://tables/users/columns`. The template matching is literal — only the `{table_name}` segment is dynamic. Extra path segments are not captured. Design URI templates to be as specific as the data they represent.

**5. stdio versus SSE for Claude Desktop.**
Claude Desktop launches stdio servers as subprocesses and manages their lifecycle. It does support remote SSE servers via the `url` field, but the behavior differs: stdio servers are restarted when Claude Desktop restarts; SSE servers are not — Claude Desktop will simply reconnect to the URL. If you need environment variable injection (secrets, paths), stdio is simpler. If you need the server to persist independently, use SSE.

## Key Takeaways

- **Three primitives:** Tool (callable, side effects OK), Resource (read-only, URI-addressed), Prompt (template, no execution).
- **Transport choice is deployment choice:** stdio for local subprocesses, HTTP/SSE for remote servers. WebSocket exists but has limited host support today.
- **Copy the skeleton, not just the snippets.** The full server skeletons above include all the wiring that is easy to get subtly wrong (name, version, transport call, error shape).
- **Validation is your contract with the model.** Pydantic and Zod schemas become the tool description the model reads. Weak schemas produce ambiguous tool calls.
- **Raise `McpError`, not bare exceptions.** Bare exceptions produce opaque `INTERNAL_ERROR` responses. Structured errors let the client (and model) reason about what went wrong.
